[Heading1]1. INTRODUCTION
[Heading2]1.1 Project Background
[Normal]The Shobhit University Gangoh Events Platform is a full-stack web application built to manage university events through a modern public website and an administrative control system. The platform is designed around the real needs of campus event publishing, student registrations, schedule visibility, university badge verification, and role-based management for admins and coordinators.
[Heading2]1.2 Problem Statement
[Normal]University events are often announced across multiple disconnected channels, which creates confusion for students and extra manual work for organizers. Event visibility, registration approval, identity verification, and participant tracking become difficult when the process is handled through ad-hoc forms or spreadsheets. A centralized system is required to present events professionally while also handling operational workflows securely.
[Heading2]1.3 Project Objectives
[Normal]- Create a public-facing university event website with a cleaner, fest-style presentation.
[Normal]- Provide secure student registration, login, email verification, and password reset.
[Normal]- Allow admins to create, update, publish, and manage events from a dedicated dashboard.
[Normal]- Support both solo and team participation models.
[Normal]- Support open events as well as university-only events protected by badge verification.
[Normal]- Enable coordinators to manage participation review for assigned events.
[Normal]- Maintain a structured backend that can be extended later with media, payments, notifications, and richer event groups.
[Heading2]1.4 Scope of the Project
[Normal]The current scope includes a React-based frontend in the client folder and an Express, Prisma, and PostgreSQL backend in the src folder. The project covers public browsing, student accounts, profile management, event registrations, admin operations, and coordinator review workflows. The system is implemented as a REST API driven application with a strongly typed frontend and backend.
[PageBreak]
[Heading1]2. PROJECT OVERVIEW
[Heading2]2.1 System Description
[Normal]This project is a university event management platform tailored for Shobhit University, Gangoh. It combines a public event discovery interface with a secure operational backend. Students can browse events, follow the schedule, maintain their profile, verify their university identity, and register for events. Administrators manage event lifecycle and users, while coordinators handle assigned event participation review.
[Heading2]2.2 Core Functional Modules
[Normal]- Public landing page and navigation
[Normal]- Events listing and event detail pages
[Normal]- Schedule page for date-based discovery
[Normal]- Student registration and login
[Normal]- Email verification using OTP
[Normal]- Password reset using OTP
[Normal]- Profile and university badge workflow
[Normal]- Solo and team event registration
[Normal]- My Events activity page
[Normal]- Admin dashboard
[Normal]- Coordinator desk
[Heading2]2.3 System Behavior
[Normal]The application follows a clean request and response model. The frontend calls backend API endpoints using Axios. The backend validates requests through Zod schemas, applies authentication and authorization middleware, performs business logic in services, and persists data using Prisma on PostgreSQL.
[PageBreak]
[Heading1]3. USER ROLES AND PERMISSIONS
[Heading2]3.1 Student
[Normal]Students are the primary end users of the platform. They can create accounts, verify their email, log in, complete their profile, submit university details for badge review, browse events, register for solo or team events, cancel registrations where allowed, view their personal event activity, and request account deletion through OTP verification.
[Heading2]3.2 Admin
[Normal]Admins have full access to the administrative workspace. They can create additional admin accounts, promote users to admin, activate or deactivate users, review university badge submissions, manage site content, create or update events, delete events, assign coordinators to specific events, and review platform data through the admin dashboard.
[Heading2]3.3 Coordinator
[Normal]A coordinator is an assigned user who manages participation review for a specific event during a defined time window. Coordinators are not a permanent standalone account type in the database. Instead, coordinator access is derived through event-specific coordinator assignments. Coordinators can access assigned events and approve or reject event registrations according to the workflow configured by admins.
[Heading2]3.4 Role Enforcement
[Normal]Role enforcement is handled by backend middleware and service-level permission checks. Admin routes require ADMIN role access. Coordinator review actions are allowed only for admins or active coordinators assigned to the specific event. Inactive accounts are blocked from authenticated flows.
[PageBreak]
[Heading1]4. SYSTEM ARCHITECTURE
[Heading2]4.1 Architectural Style
[Normal]The application uses a layered web architecture. The frontend is a client-side React application. The backend is an Express API using controllers, services, validators, utilities, and middleware. Prisma acts as the ORM and PostgreSQL is the database layer.
[Heading2]4.2 High-Level Flow
[Normal]Browser -> React frontend -> Axios API calls -> Express routes -> Controllers -> Services -> Prisma -> PostgreSQL
[Heading2]4.3 Backend Structure
[Normal]The src folder is organized into routes, controllers, services, middlewares, validators, config, utils, and type declarations. This keeps API wiring, business logic, validation, and infrastructure concerns separated cleanly.
[Heading2]4.4 Frontend Structure
[Normal]The client/src folder is organized into pages, layout components, context, hooks, shared utilities, and types. Public pages and operational pages such as admin, profile, my-events, and coordinator are kept separate while still sharing a common layout and API layer.
[Heading2]4.5 Scalability Considerations
[Normal]The codebase is structured so that features such as event groups, background images, site branding, file uploads, notifications, and payment flows can be extended without rewriting the whole application.

[Heading1]6. DATABASE DESIGN
[Heading2]6.1 Core Models
[Normal]The database schema includes User, OtpCode, UniversityBadgeLog, SiteContent, EventGroup, Event, CoordinatorAssignment, Team, TeamMember, and EventRegistration as the central models for the current platform.
[Heading2]6.2 User Model
[Normal]The User model stores account information such as email, optional phone, hashed password, full name, account role, active state, email verification state, and university profile details including university name, university email, student ID, department, course, year, and badge status.
[Heading2]6.3 OTP Model
[Normal]The OtpCode model stores hashed OTP entries used for email verification, password reset, and account deletion confirmation. Each OTP includes its purpose, status, expiration time, and audit timestamps.
[Heading2]6.4 University Verification
[Normal]UniversityBadgeLog maintains a history of badge review state transitions, including pending, verified, and rejected statuses. This creates a trackable audit trail for student badge submissions and admin reviews.
[Heading2]6.5 Event Models
[Normal]EventGroup represents a higher-level event collection and supports a future fest-style grouped-event structure. Event stores event-specific details such as title, slug, description, participation type, audience scope, status, payment requirement, registration window, venue, and scheduling data.
[Heading2]6.6 Participation Models
[Normal]Team and TeamMember support team registrations where one captain creates a team and optional members are linked to the same event. EventRegistration stores the participation record, status, payment state, review details, and the connection to either a solo user or a team registration.
[Heading2]6.7 Coordinator Model
[Normal]CoordinatorAssignment connects a user to a specific event with a start time, end time, active flag, assigning admin, and optional permission metadata. This model enables event-specific coordinator moderation without permanently changing the user’s main role.
[Heading2]6.8 Site Content Model
[Normal]SiteContent stores configurable public-facing text such as college name, festival name, hero title, subtitle, and contact details. This allows the website branding to be updated from the backend later if needed.
[PageBreak]
[Heading1]7. BACKEND DEVELOPMENT
[Heading2]7.1 API Layer
[Normal]The Express backend exposes versioned routes under /api/v1. The API is split into auth, profile, admin, events, site-content, and coordinator route groups. Each route group maps to controller functions that delegate to service-layer logic.
[Heading2]7.2 Controllers and Services
[Normal]Controllers remain thin and are responsible for request and response handling. Services contain the actual business rules such as registration validation, event creation logic, OTP validation, coordinator access checks, and university badge workflow behavior.
[Heading2]7.3 Validation
[Normal]Zod schemas are used to validate request bodies, params, and query strings before the request reaches the business logic layer. This reduces invalid input and keeps service logic focused on real business rules.
[Heading2]7.4 Error Handling
[Normal]The backend has centralized error handling for validation errors, known Prisma database errors, API errors, and unexpected internal exceptions. Clear JSON messages are returned so the frontend can show meaningful feedback.
[Heading2]7.5 Data Access
[Normal]Prisma is configured through a PostgreSQL adapter and used consistently across services. This provides typed queries, consistent relationships, and predictable database operations.

[Heading1]9. AUTHENTICATION AND ACCOUNT SECURITY
[Heading2]9.1 Registration
[Normal]Students register using full name, email, password, and optional phone number. Passwords are hashed using bcrypt before storage. A new account remains unverified until the email OTP is confirmed.
[Heading2]9.2 Email Verification
[Normal]The email verification flow issues a six-digit OTP, stores only the hashed OTP, applies expiration logic, and verifies the account after successful OTP submission. This prevents unverified accounts from logging in.
[Heading2]9.3 Login
[Normal]Login is handled through email and password. On success, the backend returns a JWT token and a user snapshot. The frontend stores the token in localStorage and uses it for authenticated API calls.
[Heading2]9.4 Password Reset
[Normal]Users can request a password reset OTP, receive a verification code, and submit a new password. Password reset OTPs are purpose-scoped and validated independently from email verification or account deletion OTPs.
[Heading2]9.5 Account Deletion
[Normal]Account deletion is protected by another OTP flow. The user requests a delete OTP and then confirms deletion by submitting the code. The backend marks the account inactive and closes active coordinator assignments rather than silently deleting records.
[Heading2]9.6 Current User Snapshot
[Normal]The auth snapshot returned by the backend includes role, effectiveRoles, email verification state, university verification state, and coordinator assignment context where applicable. This allows the frontend to render correct role-aware navigation and restricted pages.
[PageBreak]
[Heading1]10. STUDENT PORTAL
[Heading2]10.1 Public Browsing
[Normal]Students and visitors can browse published events and use the schedule page to understand upcoming activity. The event listing supports filtering by participation type, audience scope, and payment requirement.
[Heading2]10.2 Event Detail
[Normal]The event detail page shows title, description, venue, date, registration window, payment information, registration count, and the current user’s registration state if authenticated.
[Heading2]10.3 Solo Registration
[Normal]For solo events, authenticated users can register directly. The backend checks event visibility, registration window, capacity, and badge requirements before creating the registration.
[Heading2]10.4 Team Registration
[Normal]For team events, the captain creates a team and can optionally add teammate user IDs. Team size validation, duplicate participant protection, and university badge restrictions are enforced server-side.
[Heading2]10.5 University Badge Workflow
[Normal]Students can submit university information from the profile page. Admins review the submission and can verify or reject the badge. University-only events can then be restricted to verified students.
[Heading2]10.6 Personal Activity
[Normal]The My Events page and the profile page display upcoming, ongoing or recent, and past registrations so students can track their participation history in one place.
[PageBreak]
[Heading1]11. ADMIN CONSOLE
[Heading2]11.1 Admin Dashboard Purpose
[Normal]The admin dashboard is the main operational panel for managing users and events. It is separated from the public browsing experience and is intended for internal platform control rather than public content browsing.
[Heading2]11.2 User Management
[Normal]Admins can list users, promote eligible users to admin, create a new admin account directly, activate or deactivate user accounts, and review university badge submissions. Inactive users are blocked from authentication-protected routes.
[Heading2]11.3 Event Management
[Normal]Admins can create, update, publish, cancel, complete, and delete events. The current frontend event form is intentionally simpler, and the backend supports it by assigning such events to a default event group when no explicit group is provided.
[Heading2]11.4 Event Publication Control
[Normal]The admin chooses whether an event is draft, published, cancelled, or completed. Only published events are visible publicly and can accept registrations within the configured registration window.
[Heading2]11.5 Future-Ready Features
[Normal]The backend also contains support structures for site content management, event groups, media-ready event fields, and coordinator assignment. These make it possible to evolve the platform further without rebuilding the data model.
[PageBreak]
[Heading1]12. COORDINATOR WORKFLOW
[Heading2]12.1 Coordinator Assignment
[Normal]Admins can assign a user as a coordinator for a specific event and define the start and end time of that assignment. This makes the coordinator role temporary, event-specific, and auditable.
[Heading2]12.2 Coordinator Access
[Normal]When a coordinator is active for a specific event, the backend includes that assignment in the authenticated user snapshot. The frontend can then expose coordinator-specific access, such as the coordinator desk.
[Heading2]12.3 Participation Review
[Normal]Coordinators can retrieve assigned events and review registrations for those events through coordinator endpoints. Registrations can be confirmed or rejected, and a review note can be stored for auditability.
[Heading2]12.4 Why This Design
[Normal]This design avoids turning coordinator into a permanent top-level user role. It keeps the system flexible and better aligned with how event staffing often changes over time in universities.

[Heading1]14. SECURITY IMPLEMENTATION
[Heading2]14.1 Password Security
[Normal]Passwords are hashed using bcrypt before being stored in the database. Plain text passwords are never persisted.
[Heading2]14.2 Token Security
[Normal]Authenticated access is protected through JWT bearer tokens. The backend validates the token and resolves the active user on each protected request.
[Heading2]14.3 OTP Handling
[Normal]OTP values are hashed before storage, time-limited, purpose-specific, and status-tracked. Invalid submissions increase attempt counters, and used or expired OTPs are marked accordingly.
[Heading2]14.4 Validation and Input Safety
[Normal]Zod validation blocks malformed requests before they reach business logic. Prisma parameterized access protects the data layer from direct query injection risks.
[Heading2]14.5 Role and Account Controls
[Normal]Admin-only actions are protected by role middleware. Event review actions require either admin role or a valid active coordinator assignment. Inactive users are denied authenticated access.
[Heading2]14.6 Error Handling
[Normal]The backend returns structured JSON messages for validation failures, API errors, Prisma request failures, and unexpected server errors. This gives the frontend enough information to display useful feedback while keeping the architecture maintainable.
